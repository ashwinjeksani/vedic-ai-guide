# Deploying Sanatana on Render (with passkey auth + allowlist)

This adds a real backend to your Vite app: passkey (WebAuthn) sign-in, a
server-enforced free-question limit, and an admin allowlist you control.

## What's in this bundle

```
vedic-deploy/
├── render.yaml                  # one-click Render blueprint
├── DEPLOY.md                    # this file
├── frontend-auth-snippet.js     # drop into your frontend as src/auth.js
└── server/
    ├── package.json
    ├── index.js                 # express entry: routes + static hosting
    ├── db.js                    # SQLite schema (users, credentials, request_log)
    ├── auth.js                  # WebAuthn register/authenticate + sessions
    └── routes.js                # gated /api/chat proxy + admin routes
```

## How the access model works (all server-side, nothing hidden)

- **Anyone can register** a passkey with a username.
- The **first account to register becomes admin** (or set `ADMIN_USERNAME` to
  claim a specific name). Admins are auto-allowlisted.
- **Pending (non-allowlisted) users** get `FREE_LIMIT` questions (default: 10),
  then `/api/chat` returns `403 limit_reached`.
- **Allowlisted users** are unlimited.
- The **admin approves people** from `/api/admin/users` — flip a user to
  `allowlisted`. This is a normal, auditable admin role: the check is
  `user.role === 'admin'`, visible in `routes.js`, logged like any request.

The quota and allowlist live in the backend precisely so they can't be
bypassed from the browser (the same reason your API keys must stay server-side).

## Repo layout expected by render.yaml

Put your existing Vite app in a `frontend/` folder at the repo root, and this
`server/` folder alongside it:

```
your-repo/
├── frontend/       # your current Vite app (index.html, src/, vite.config.js, vedic-guide.jsx)
├── server/         # from this bundle
├── render.yaml     # from this bundle
```

If your Vite app currently lives at the repo root, just move it into
`frontend/`. You can also delete the old `/api/chat` middleware from
`frontend/vite.config.js` — the real proxy now lives in `server/routes.js`.
(Keep it only if you still want local dev without the Node server.)

## Frontend changes

1. `cd frontend && npm install @simplewebauthn/browser`
2. Copy `frontend-auth-snippet.js` to `frontend/src/auth.js`.
3. In your component, gate the composer: call `me()` on load; if there's no
   user, show a "Register a passkey" / "Sign in" pair of buttons that call
   `register(username)` / `login(username)`. When `/api/chat` returns
   `limit_reached`, show a "request access" message. (Your existing fetch to
   `/api/chat` already matches the backend — same `provider` / `model` /
   `system` / `messages` shape.)
4. Optional: a small admin page that calls `adminListUsers()` and
   `adminSetStatus(userId, 'allowlisted')`, shown only when `me().role === 'admin'`.

## Deploy steps

1. Push the repo (with `frontend/`, `server/`, `render.yaml`) to GitHub.
2. In Render: **New → Blueprint**, point it at the repo. Render reads
   `render.yaml` and creates the web service + 1 GB persistent disk.
3. In the service's **Environment** tab, fill the `sync:false` vars:
   - `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` — your keys
   - `RP_NAME` — e.g. `Sanatana`
   - `ADMIN_USERNAME` — the username you'll register first
   - `RP_ID` and `ORIGIN` — see the domain step below.
   `SESSION_SECRET` is auto-generated by Render.
4. First deploy will use the temporary `onrender.com` URL. For that to work
   before your domain is attached, set (temporarily):
   - `RP_ID = sanatana.onrender.com` (your actual Render subdomain, no scheme)
   - `ORIGIN = https://sanatana.onrender.com`
5. Open the site, register your admin username → you're the admin.

## Attaching your purchased domain

1. Render service → **Settings → Custom Domains → Add** `sanatana.yourdomain.com`.
2. At your registrar, add the CNAME Render shows you. HTTPS is issued automatically.
3. **Important for passkeys:** update the env vars to the real domain and redeploy:
   - `RP_ID = sanatana.yourdomain.com` (host only — no `https://`, no port)
   - `ORIGIN = https://sanatana.yourdomain.com`
   WebAuthn ties credentials to the exact `RP_ID`; if it doesn't match the
   domain in the address bar, sign-in silently fails. Passkeys registered
   under the temporary `onrender.com` RP_ID won't carry over — register again
   under the final domain.

## Notes & caveats

- **Cold starts:** the free plan spins down after ~15 min idle (30–60 s wake).
  Upgrade the service to `starter` in `render.yaml` (`plan: starter`) to keep it warm.
- **SQLite on a disk** is fine for a single instance. If you ever scale to
  multiple instances, move to Render Postgres (the schema ports directly).
- **Sessions** here are a signed httpOnly cookie carrying the user id. For a
  stricter setup, add a server-side session table and store only a random
  token in the cookie.
- **Free-tier leakage:** because the free question requires registration, the
  counter attaches to a real passkey — much harder to reset than a cookie. A
  determined person could still register multiple usernames; that's acceptable
  since the *valuable* access is the admin-approved allowlist.
